# 검증 패널 (verification.py) 평가 루브릭

## 무엇을 재는가

검증 패널이 "잘 동작한다" = **통계 탐지가 낸 이상 중, 조치가 필요 없는 오탐은 강등하고
실제 조치가 필요한 이상은 유지**한다. 즉 세 심사관의 다수결이 도메인적으로 올바른
판정(진짜/오탐)과 일치한다.

패널은 LLM 3명(배포/재시작 회의론자, 지속성 심사관, 근거 심사관)이 독립적으로
`real | false_positive | uncertain`을 투표하고, 과반(2/3)이 `false_positive`면
판정을 한 단계 강등한다. 이 강등 결정이 정답과 맞는지가 핵심이다.

## 정답을 정한 원칙 (구현 독립)

정답(`expected.downgrade`)은 **현재 코드가 무엇을 내는지 보지 않고** 도메인 규칙으로 정했다:

- **오탐 신호 → 강등 기대 (`downgrade: true`)**: 기준선 표본 부족(n<5), 단일 1분 버킷의
  순간 스파이크, 극소 총량에서의 높은 비율(예: 2건 중 1건), 재시작·확장 직후 워밍업.
- **실제 문제 → 유지 기대 (`downgrade: false`)**: 충분한 표본(n≥30) + 지속성 + 유의미한
  총량. 오류/포화/지연/스케줄러 실패가 명백하고 지속되는 경우.
- **경계(`borderline: true`)**: 어느 쪽도 명백히 틀리지 않은 케이스. 정확도 축에서 기대
  방향을 유지로 두되, 반대가 나와도 "실패"로 세지 않고 별도 표기한다.

## 평가 축

### A. 판정 정확도 (0/1) — 자동 채점
패널의 `downgrade` 결정이 `expected.downgrade`와 일치하면 1, 아니면 0.
경계 케이스는 이 축의 합산에서 제외하고 별도로 보고한다(정답이 느슨하므로).

- **1**: downgrade == expected.downgrade
- **0**: 불일치 (오탐을 유지했거나, 진짜를 강등했거나)

가장 중요한 축. 특히 **진짜 이상을 강등(false negative)** 은 실제 장애를 놓치는 것이라
오탐을 유지하는 것보다 위험하다 — 리포트에서 두 방향을 구분해 센다.

### B. 심사관 합의 품질 (0-3) — 자동 채점
정답 방향으로 투표한 심사관 수.
- 오탐 케이스: `false_positive` 투표 수
- 진짜 케이스: `real` 또는 `uncertain` 투표 수 (= 3 − false_positive 투표 수)

`min_correct_lenses`는 이 축의 합격선(느슨한 기대치)이다. A가 맞아도 3명이 아슬아슬하게
갈렸는지(2:1) 압도적이었는지(3:0)를 구분해 패널의 확신도를 본다.

### C. 일관성 (0/1) — 자동 채점 (--repeat 옵션)
같은 케이스를 N회(기본 3회) 반복했을 때 `downgrade` 결정이 매번 동일하면 1.
`temperature=0`이지만 LLM은 완전 결정론적이 아니므로, 판정이 흔들리면 신뢰할 수 없다.
비용이 있으므로 `--repeat N`을 줄 때만 채점한다.

### D. 근거 타당성 (0-2) — **자동 채점 불가, 수동/LLM-judge**
각 심사관의 `reason`이 실제 입력 근거(표본 수, 지속성, 총량, 재시작 등)에 기반했는가,
아니면 지어냈는가. 텍스트의 사실 정합성이라 규칙 기반 자동 채점이 불가능하다.
`run.py`가 각 케이스의 reason을 덤프하므로, 사람이 아래 기준으로 채점한다:

- **2**: 모든 reason이 입력에 실재하는 근거를 정확히 인용
- **1**: 일부만 근거 기반, 나머지는 모호하거나 일반론
- **0**: 입력에 없는 사실을 지어냄(환각), 또는 근거와 모순

수동 채점란: `evals/manual_scores.md` (없으면 생성).

## 채점 실행

```bash
cd ai_service
set -a; source ../.env; set +a          # OPENAI_API_KEY 필요 (ES는 불필요)
python evals/run.py                       # A, B 채점 (케이스당 1회)
python evals/run.py --repeat 3            # A, B, C 채점 (일관성 포함, 비용↑)
```

## 케이스 구성

| 범주 | 개수 | 정답 |
|---|---|---|
| real (진짜 이상) | 5 | 유지 (downgrade=false) |
| false_positive (오탐) | 5 | 강등 (downgrade=true) |
| borderline (경계) | 2 | 유지 기대, 반대도 허용 |

실제 프로덕션 이상 데이터가 거의 없어(대부분 정상, 이상은 데모 합성), 케이스는
데모 시나리오(api-failure/traffic-spike/scheduler-failure)의 지표 형태와 경계 케이스로
구성했다. 각 케이스는 실제 detection.py가 만드는 result/metrics/baseline 형태를 따른다.
