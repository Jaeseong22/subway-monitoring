LLM은 완전 결정론적이 아니므로 실행마다 소폭 흔들릴 수 있다.

### 드러난 결함 (eval이 잡은 것)

오탐 판별은 완벽(5/5)하나, **진짜 이상 2건을 강등**한다. 근거 덤프에서 원인 확인:

1. **인스턴스 수 오판**: 세 심사관이 `instance_count: 1`을 보고 "단일 인스턴스라
   신뢰할 수 없다 / 일시적 부하일 수 있다"고 오탐 판정. 인스턴스 수는 이상 판정과
   무관한데 LLM이 근거로 지어냄.
   > 배포/재시작 회의론자: "인스턴스가 1개만 가동 중이며 ... 일시적인 현상일 수 있다"

2. **표본 수 오판**: `real_latency_sustained`는 기준선 40건인데 근거 심사관이
   "40건으로 적어 신뢰할 수 없다"고 판정. 40건은 충분하다(min_samples=5).

false negative는 실제 장애를 놓치는 것이라 오탐 유지보다 위험하다. 프롬프트가 심사관을
과도하게 회의적으로 만든 결과.

## 실행 #1 — 2026-07-22 (프롬프트 가드 추가 후)

`verification.py`의 `_GUARDRAILS` 추가: 심사관이 (1) 인스턴스 수, (2) 30건 이상
baseline을 오탐 근거로 삼지 못하게 명시. "3배 이상 + 지속"은 인스턴스 수와 무관하게
진짜 문제라고 못박음.

| 축 | #0 (전) | #1 (후) | 변화 |
|---|---|---|---|
| A. 판정 정확도 (경계 제외) | 8/10 (80%) | **10/10 (100%)** | +20%p |
| — false negative (위험) | 2건 | **0건** | 해결 |
| — false positive | 0건 | 0건 | 유지 |
| B. 심사관 합의 | 22/36 (61%) | **33/36 (92%)** | +31%p |
| C. 일관성 (3회 반복) | 미측정 | **12/12 안정** | |
| D. 근거 타당성 | 미채점 | 미채점 | |

프롬프트 가드 하나로 진짜 이상을 강등하던 false negative가 사라졌고, 오탐 판별(5/5)은
그대로 유지됐다. eval이 없었으면 "세 에이전트 잘 돈다"는 직감에 묻혔을 결함이다.

### 재현

```bash
cd ai_service
set -a; source ../.env; set +a
python evals/run.py                 # A, B
python evals/run.py --repeat 3      # + C(일관성)

```
